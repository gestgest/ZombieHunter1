# 좀비 헌터 개발로그 (2026.07.06 ~ 07.12)

이번 주는 **무한맵 위의 큰 오브젝트 — 마을(POI)** 이다.
나무나 돌 같은 장애물은 청크마다 랜덤으로 흩뿌리면 되지만, 마을처럼 **큰 영역을 차지하는
오브젝트**는 "그 자리에 장애물이 안 생기게" 해야 한다. 이걸 POI 스켈레톤으로 풀었다.

> **핵심 원칙: 겹침은 검사가 아니라 순서(결정론)로 해결한다.**
> "일단 만들고 겹치면 지운다"가 아니라, 애초에 만드는 루프에 진입을 안 한다.

---

## 1. POI 스켈레톤 — 구조

청크를 **리전(8×8 청크 = 160m)** 단위로 묶고, 리전마다 시드 해시로 최대 1개의 POI를 정한다.
POI 발자국 안의 청크는 전용 바닥을 깔고 장애물을 만들지 않는다. (콘텐츠는 후속 단계)

```cpp
/** POI(특수 지역) 종류 */
enum class EPOIType : uint8
{
    Village,        // 마을 (발판/NPC 등 — 후속 단계에서 채움)
    ZombieVillage,  // 좀비마을 (적 밀집 + 보상 — 후속 단계에서 채움)
};

/** 리전 하나의 POI 정보. 시드 해시로만 결정되므로 어느 청크에서 계산해도 항상 같은 답이 나온다. */
struct FPOIInfo
{
    bool bHasPOI = false;
    FIntPoint CenterChunk = FIntPoint::ZeroValue;  // POI 중심 청크 (전역 청크 좌표)
    EPOIType Type = EPOIType::Village;
    bool bIsCenter = false;  // GetPOIAtChunk가 채움: 질의한 청크가 중심인지 (콘텐츠 스폰은 중심에서 한 번만)
};
```

### 설정 변수

```cpp
/** 리전 한 변의 청크 수. 리전마다 최대 1개의 POI가 배치된다. (8 × ChunkSize 2000 = 160m마다) */
UPROPERTY(EditAnywhere, BlueprintReadOnly, Category = "Map|POI", meta = (ClampMin = "2"))
int32 RegionSizeInChunks = 8;

/** POI 한 변의 청크 수 (예: 3이면 3×3 청크 = 60×60m). 홀수 권장 — 짝수는 아래 홀수로 내림.
 *  리전보다 크게 잡으면 리전에 들어가는 최대 크기로 자동 축소된다. */
UPROPERTY(EditAnywhere, BlueprintReadOnly, Category = "Map|POI", meta = (ClampMin = "1"))
int32 POISizeInChunks = 3;

/** 리전에 POI가 생길 확률(0~1). 시작 리전(0,0)은 이 값과 무관하게 항상 마을이 보장된다. */
UPROPERTY(...) float POIChance = 0.8f;

/** POI가 마을일 확률(나머지는 좀비마을). */
UPROPERTY(...) float VillageRatio = 0.6f;

/** 마을/좀비마을 바닥 머티리얼 (기본: 파랑 / 어두운 그리드 — 에디터 작업 없이 바로 구분) */
UPROPERTY(...) TObjectPtr<UMaterialInterface> VillageFloorMaterial;
UPROPERTY(...) TObjectPtr<UMaterialInterface> ZombieVillageFloorMaterial;

/** 켜면 POI 경계 박스(마을=초록, 좀비마을=빨강)와 로그 */
UPROPERTY(...) bool bDebugDrawPOI = true;
```

머티리얼은 생성자에서 `ConstructorHelpers::FObjectFinder`로 기본 지정해서,
빌드 직후 에디터 작업 없이 바로 눈으로 확인 가능하게 했다 (걸어다니다 보면 색 다른 땅 발견).

### 판정 흐름

```
Tick (0.25초마다)
 └─ UpdateChunks              "플레이어 주변 7×7 청크 유지"
     └─ GenerateChunk
         └─ GetPOIAtChunk         "이 청크가 POI 발자국 안이야?"
             ├─ ChunkToRegion         "이 청크는 어느 리전 소속?"
             └─ GetPOIForRegion       "그 리전의 POI는 어디에 뭐가 있지?"
```

전부 **시드 해시 순수 계산**이다. 물리 검사·스폰·청크 간 통신이 없으므로,
청크가 어떤 순서로 로드/언로드돼도 항상 같은 답이 나온다.

## 2. POI 크기 — 발자국(Footprint)

처음 스켈레톤은 POI가 무조건 **1청크(20m)** 였다(언리얼은 1유닛=1cm라 2000유닛=20m).
마을이 20m면 집 두어 채 크기라 너무 작다. 그래서 POI가 여러 청크를 차지하도록 확장했다.

> **POI = 중심 청크 + 반경 R.** 발자국 한 변은 항상 `2R+1` — 무조건 홀수다.
> (짝수면 "중심 청크"가 존재할 수 없어서 이 구조로 표현이 안 된다)

### 반경 계산은 단일 창구로

반경이 필요한 곳이 세 군데다(리전의 중심 뽑기, 청크의 발자국 판정, 디버그 박스).
각자 계산하면 한 곳만 어긋나도 마을이 잘리므로, 함수 하나로 모았다.

```cpp
//POI 발자국 반경(청크). 한 변 = 2R+1이 리전을 못 넘게 제한
int32 AInfiniteMapGenerator::GetPOIRadiusInChunks() const
{
    const int32 RegionSize = FMath::Max(2, RegionSizeInChunks); //음수나 0 나누기 방지용
    const int32 Radius = (FMath::Max(1, POISizeInChunks) - 1) / 2; // 짝수는 홀수로 내림: 4 → 3×3
    return FMath::Min(Radius, (RegionSize - 1) / 2);
}
```

한 줄씩 전부 보정이다. UPROPERTY는 에디터/BP에서 아무 값이나 들어올 수 있기 때문이다
(`ClampMin` 메타는 Details 패널 **UI만** 막아준다 — 코드/BP 경로는 못 막는다).

| 입력 | 보정 없으면 | 보정 후 |
|---|---|---|
| `RegionSizeInChunks = 0` | 0으로 나누기 | 2로 취급 |
| `POISizeInChunks = -3` | 반경 음수 → POI 전부 소멸 | 반경 0 → 1청크 POI |
| `POISizeInChunks = 99` (리전 8) | 중심 뽑기 범위 붕괴 (`RandRange(49, -42)`) | 리전 최대(7×7)로 축소 |

> **삽질 기록**: 처음에 `Size / 2`로 썼다가 4를 넣으면 3×3이 아니라 **5×5가 되는(올림)**
> 버그가 있었다. `(Size - 1) / 2`가 진짜 내림. 주석과 코드가 다르면 십중팔구 버그다.

### 판정: "중심인가?" → "발자국 안인가?"

`GetPOIAtChunk`가 정확히 일치가 아니라 **Chebyshev 거리(체스판 거리) ≤ R**을 본다.

```cpp
bool AInfiniteMapGenerator::GetPOIAtChunk(const FIntPoint& ChunkCoord, FPOIInfo& OutInfo) const
{
    // 중심을 리전 안쪽으로 클램프해 발자국이 리전을 못 벗어나므로, 자기 리전만 검사하면 된다
    const FPOIInfo Info = GetPOIForRegion(ChunkToRegion(ChunkCoord));
    if (!Info.bHasPOI)
    {
        return false;
    }

    const int32 Radius = GetPOIRadiusInChunks();
    if (FMath::Abs(ChunkCoord.X - Info.CenterChunk.X) <= Radius &&
        FMath::Abs(ChunkCoord.Y - Info.CenterChunk.Y) <= Radius)
    {
        OutInfo = Info;
        OutInfo.bIsCenter = (ChunkCoord == Info.CenterChunk);
        return true;
    }
    return false;
}
```

이 한 곳만 바꾸면 나머지는 공짜다. 발자국 9개(3×3) 청크가 **각자 독립적으로**
"나는 마을 소속" 판정을 내리고 알아서 바닥을 바꾸고 장애물을 끈다.
"중심 청크가 주변 청크의 장애물을 지워준다" 같은 청크 간 통신이 전혀 필요 없다.

`bIsCenter`는 나중에 발판/NPC 같은 핵심 콘텐츠를 **중심 청크에서 딱 한 번만**
스폰할 자리다 (발자국 전 청크에서 스폰하면 9개씩 생긴다).

### 리전 경계 — "잘리는" 게 아니라 "못 넘게"

발자국이 리전 경계를 넘으면 문제가 된다. 옆 리전 청크는 **자기 리전의 POI만** 계산하므로,
남의 리전에서 넘어온 발자국을 모르고 일반 청크로 생성 → 마을이 한 줄 잘린다.

해결은 모든 청크가 주변 3×3 리전을 검사(계산 9배)하는 게 아니라,
**중심을 애초에 리전 안쪽 `[R, Size-1-R]`에서만 뽑는 것.** 8×8 리전에 R=1이면
중심은 로컬 1~6 — 발자국 끝이 딱 경계(0, 7)까지만 닿고 절대 못 넘는다.

```cpp
// GetPOIForRegion 내부
// 리전 내 POI 중심 위치. 발자국(2R+1)²이 리전 밖으로 삐져나가지 않게 중심을 [R, Size-1-R]에서만 뽑는다.
const int32 Radius = GetPOIRadiusInChunks();
const int32 MaxCell = Size - 1 - Radius;
// 시작 리전은 발자국 가장자리(중심-R)가 플레이어 시작 청크(0,0)에서 최소 2청크 떨어지게
const int32 MinCell = FMath::Min(bStartRegion ? (Radius + 2) : Radius, MaxCell);
const int32 LocalX = Stream.RandRange(MinCell, MaxCell);
const int32 LocalY = Stream.RandRange(MinCell, MaxCell);
Info.CenterChunk = FIntPoint(RegionCoord.X * Size + LocalX, RegionCoord.Y * Size + LocalY);
```

### 스타팅 마을 보정

시작 리전(0,0)은 `POIChance`와 무관하게 **항상 마을**이다 — 첫 마을 발견을 보장.

```cpp
const bool bStartRegion = (RegionCoord == FIntPoint::ZeroValue);
if (!bStartRegion && Stream.FRand() >= POIChance)
{
    return Info; // 이 리전엔 POI 없음
}
Info.bHasPOI = true;
```

거리 보정도 발자국에 맞춰 바뀌었다: 예전엔 "중심이 시작점에서 2청크"였는데,
발자국이 생기니 **가장자리 기준**으로 떨어뜨려야 해서 `R + 2`가 됐다
(중심만 멀리 보내도 발자국이 스폰 지점을 덮을 수 있으니까).

## 3. GenerateChunk에서 POI가 하는 일 — 헷갈렸던 것들

`GenerateChunk` 맨 앞에서 판정 하나를 하고, 그 결과를 **세 곳**이 나눠 쓴다.

```cpp
FPOIInfo POI;
const bool bIsPOIChunk = GetPOIAtChunk(Coord, POI);   // ← 판정은 여기 한 번

// (1) 바닥 머티리얼 결정   → if (bIsPOIChunk)  { FloorMat = POIMat; }
// (2) 바닥 스폰 (한 번만!)  → if (FloorMesh)    { SpawnMeshActor(..., FloorMat); }
// (3) 장애물 통째로 스킵    → if (!bIsPOIChunk) { 장애물 루프 }
// (4) 디버그 박스 (스폰 없음)→ if (bIsPOIChunk && bDebugDrawPOI && POI.bIsCenter) { DrawDebugBox }
```

**Q. 바닥 스폰을 `if (bIsPOIChunk)` 안에 넣으면 안 되나?**
안 된다. 바닥은 POI든 아니든 **모든 청크**에 깔려야 한다. 넣으면 맵의 99%가 허공이 된다.
분기 간 차이가 머티리얼 하나뿐이라, **달라지는 것(머티리얼)만 변수로 먼저 결정하고
공통 동작(스폰)은 한 벌만** 두는 구조다(결정/실행 분리). if/else로 스폰을 복제하면
나중에 바닥 계산을 고칠 때 두 군데를 똑같이 고쳐야 하는 함정이 생긴다.

**Q. 그럼 마을 깔기가 두 번인가? (바닥 블록 + 디버그 블록)**
아니다. 디버그 블록은 **아무것도 스폰하지 않는다.** `DrawDebugBox`는 액터가 아니라
화면에 와이어프레임 선을 그리는 시각화일 뿐이다 — 충돌 없음, 120초 후 소멸,
Shipping 빌드에선 아예 안 나옴. `bDebugDrawPOI`를 꺼도 마을 바닥은 멀쩡히 깔린다.
박스는 중심 청크가 생성될 때 발자국 전체 크기로 **한 번만** 그린다:

```cpp
const int32 FootprintSize = GetPOIRadiusInChunks() * 2 + 1;
const float HalfExtent = ChunkSize * FootprintSize * 0.5f;  // DrawDebugBox는 반크기를 받는다
// 로그: [POI] Village generated at chunk (11, 6), size 3x3 chunks
```

**Q. 장애물이 안 생기게 하는 코드는 어디?**
`if (!bIsPOIChunk && ObstacleMeshes.Num() > 0)` — 조건문 하나가 전부다.
"지우는" 코드는 없다. 만드는 루프에 진입을 안 하니 마을 자리에 장애물이 존재한 적조차 없다.

## 4. 크기 감각 정리

| 단위 | 크기 |
|---|---|
| 청크 (`ChunkSize 2000`) | 20m × 20m |
| POI 3×3 (기본) | 60m × 60m (축구장 폭 정도) |
| 리전 8×8 | 160m × 160m — 이 안에 POI 최대 1개 |
| 뷰 반경 3청크 (7×7 유지) | 140m × 140m |

마을 크기만 조절하고 싶으면 `POISizeInChunks`만 건드린다.
`ChunkSize`를 바꾸면 장애물 밀도·뷰 거리·NavMesh 범위가 다 같이 영향받는다.

## 5. 개선할 점 / 남은 것

- [ ] **분포 편향**: 중심이 리전 가장자리 셀에 못 오므로 POI가 리전 중앙 쪽으로 몰린다.
      지금(8×8에 3×3)은 체감 없지만, POI를 7×7까지 키우면 중심 후보가 3~4뿐 —
      그땐 `RegionSizeInChunks`도 같이 키워야 한다.
- [ ] **디버그 박스 타이밍**: 발자국 가장자리 청크가 먼저 로드되면 바닥색은 보이는데
      박스는 중심 청크가 로드될 때까지 안 뜬다 (디버그용이라 감수).
- [ ] **발자국 내부 구분 없음**: 지금은 9청크가 전부 같은 바닥. 나중에 가장자리(담장/입구) /
      중앙(발판/NPC) 레이아웃을 나눌 때 `bIsCenter` + 중심으로부터의 로컬 오프셋을 쓰면 된다.
- [ ] **상태 영속(`FPOIState`)**: 별개 과제 그대로 — 발자국이 커져도 상태 키는 여전히
      "리전당 POI 1개"라 리전 좌표(또는 중심 청크) 하나면 된다.
- [ ] 시드 소비 순서는 유지했지만 중심 뽑기 범위가 바뀌어서, 같은 시드라도
      스켈레톤 시절과 POI 위치가 달라질 수 있다 (저장 상태가 없으니 실해는 없음).

다음 단계는 기획 문서(poi-event-design.md)의 순서대로: **POI 상태 영속 → 마을 v1
(HealZone + 기존 발판 + 적 스폰 제외) → 좀비마을.**
