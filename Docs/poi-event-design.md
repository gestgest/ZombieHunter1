> 2026-07-07 작성. progress-log의 "무한맵에 걸맞는 이벤트" 항목(마을/좀비마을)의 설계 문서.

## 1. 핵심 컨셉: POI 청크 (Point of Interest)

무한맵 위에 **시드 기반으로 결정적으로 배치되는 특수 지역**. 같은 시드면 같은 자리에
항상 같은 마을이 생기므로, 플레이어가 떠났다가 돌아와도 그 마을이 그 자리에 있다.
"탐험하다 보면 마을을 발견한다"가 무한맵의 핵심 재미 포인트가 된다.

### 배치 규칙 (결정적 생성)

- 청크를 **리전(Region)** 단위로 묶는다. 예: 8×8 청크 = 1리전 (160m × 160m)
- `Hash(GlobalSeed, RegionCoord)` 로 리전마다:
  - POI 존재 여부 (예: 80% 확률)
  - 리전 내 POI 중심 청크 좌표
  - POI 타입 (마을 60% / 좀비마을 40%)
  를 결정한다. → 어떤 청크가 로드되든 "내 리전의 POI가 어디고 뭔지"를 독립적으로 계산 가능
- **시작 지점 보정**: 플레이어 시작 리전(0,0)은 무조건 마을 + 시작점에서 몇 청크 떨어진 위치 보장
  (너무 가까우면 긴장감 없고, 너무 멀면 발견 못 함)

### 멀티 청크 POI (처음부터 안전하게 설계)

마을이 1청크(20m)보다 커도 되도록, 각 청크는 생성 시 이렇게 판단한다:

```
내 리전의 POI 중심 = POICenter (해시로 계산, 어느 청크든 동일한 답)
내가 POICenter로부터 반경 R청크 이내인가?
  → 예: 마을풍 바닥(색/메시 다름) 생성, 장애물 생성 안 함
  → 내가 정중앙 청크인가? → 발판/NPC 등 핵심 액터 스폰
```

이러면 청크가 어떤 순서로 로드/언로드돼도 마을이 항상 일관되게 생긴다.
v1은 반경 R=0 (1청크 마을)로 시작하고, 나중에 R만 키우면 3×3 마을이 된다.

### 겹침(장애물 vs POI) 처리 — 검사가 아니라 순서로 해결

생성이 청크 단위 + 결정론이라 물리 오버랩 검사가 필요 없다:

- `GenerateChunk` 맨 앞에서 POI 판정(해시 계산) → **마을 청크면 장애물 루프를 아예 스킵**하고
  마을 생성 경로를 탄다. 큐브/실린더/콘을 마을이 "대체" — 같은 청크에 공존하지 않는다.
- **마을 내부 배치는 고정 레이아웃 템플릿**: 발판/NPC 위치를 청크 로컬 오프셋으로 손으로 짜두고,
  템플릿 2~3개 중 해시로 선택. 랜덤 산포가 아니므로 내부 겹침도 원천 차단 +
  마을이 "디자인된 공간"처럼 보임.
- (선택) 일반 청크의 장애물끼리 겹침도 없애려면: 청크를 4×4 서브그리드 셀로 나누고
  셀을 셔플해 셀당 최대 1개 배치. 재추첨(rejection) 루프보다 단순하고 실패 케이스가 없음.
- 주의: POI 분기를 넣으면 Stream 소비 순서가 바뀌어 기존 청크들의 장애물 배치가
  지금과 달라진다(저장 상태가 없으니 실해는 없음).

## 2. 핵심 기술 과제: 상태 영속

청크 언로드 시 액터가 전부 Destroy된다. 시드 덕에 **재생성은 공짜**지만,
**변한 상태**(발판 진행도, 강화 구매 여부, NPC 생사, 퀘스트 진행)는 날아간다.

→ `TMap<FIntPoint, FPOIState>` 를 **GameInstance**(레벨 리로드에도 생존)에 보관.
청크 언로드 시 상태 저장, 재생성 시 복원. 디스크 세이브는 이번 범위 아님.

```
FPOIState (초안)
- 발판별 PaidMoney / bConsumed   (일회성 스폰존이 이미 쓰였는지 등)
- NPC 생사 (v3)
- 퀘스트 상태 (v3)
```

## 3. 마을 구성 — 단계별로 쪼개기

progress-log의 마을 항목을 전부 한 번에 하면 개빡세니(원문 존중) 단계로 나눈다:

| 단계 | 내용 | 비고 |
|---|---|---|
| **v1** | 구별되는 바닥 + 회복 발판(`AHealZone` 신규) + 무기 강화 발판(기존 `AWeaponUpgradeZone` 재활용) + 마을 안 적 스폰 금지 | 발판은 `AMoneyPadZone` 서브클래스라 거의 공짜 |
| **v2** | 직업 스폰존(랜덤 직업, `bOneShot` 활용 — 이미 있는 기능!) + 무기 상점 | |
| **v3** | NPC(퀘스트 주는/공격하는, 적에게 공격당함) | 대화/퀘스트 UI가 필요한 **별도 큰 기획** |
| **v4** | 좀비 침공 이벤트 (마을에 머물면 확률 발동) | 마을=안전지대의 긴장 요소 |

- **회복 유료 여부** → `AMoneyPadZone` 재활용하면 자연히 유료(밟고 서 있으면 돈 소모하며 회복).
  `MoneyPerPayment`로 가격 조절. 답이 이미 시스템 안에 있음.
- **마을 안 적 스폰 금지**: `FindReachablePointInRing` 결과가 마을 반경 안이면 재시도.
  단, v4 침공 때는 이 규칙을 의도적으로 깬다.

## 4. 좀비마을

- 장애물 빽빽 + 적 다수 배치(스폰 링과 별개로 POI 전용 스폰) + 보상: 돈 다발, 1% 확률 무기
- 마을보다 구현이 단순 (발판/NPC 없음, 지형+적+드랍뿐) → 마을 v1 다음에 하면 금방

## 5. 구현 순서 제안

1. **POI 스켈레톤** ← 제일 먼저
   - `AInfiniteMapGenerator`에 리전/POI 선정 로직 + 마을 자리에 색 다른 바닥 깔기
   - **에디터 작업 없이 빌드 직후 바로 눈으로 확인 가능** (걸어다니다 보면 색 다른 땅 발견)
2. **POI 상태 영속** — GameInstance에 FPOIState, 발판 스폰/복원 배선
3. **마을 v1** — HealZone 신규 + 기존 발판 스폰 + 적 스폰 제외
   - 에디터 작업: BP_HealZone 생성, 생성기 Details에 발판 BP 클래스 지정 (소량)
4. **좀비마을**
5. **v2(스폰존/상점) → v4(침공) → v3(NPC/퀘스트)** — NPC는 별도 기획 후
