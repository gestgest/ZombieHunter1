# 좀비 헌터 개발로그 (2026.06.22 ~ 2026.06.28)

지난주에 세운 **"전투의 뼈대"**를 이번 주엔 클래스 구조 관점에서 다시 정리했다.
새 직업/캐릭터를 붙이면서 "이건 어디에 둬야 하지?"가 자꾸 헷갈려서,
**전투 코드를 두 개의 축으로 쪼갠 설계**를 문서로 못박아 둔다.

> **공격 방식은 "직업"(`UJobComponent`)이, 체력·죽음은 "몸"(`ACombatCharacter`)이 가진다.**
> 그래서 플레이어·동료·적이 코드를 공유한다.

---

## 1. 직업 구분 — 검사 / 궁수 / 마법사 / 힐러

직업을 **검사, 궁수, 마법사, 힐러** 4개로 나눴다(와우식). 시작 시 1개를 골라 고정한다.

핵심은 직업을 "캐릭터"가 아니라 **"컴포넌트"로 만든 것**이다(`UJobComponent`).
이동/조준은 소유 캐릭터가 하고, **공격 방식만** 컴포넌트가 책임진다. 소유자가 `ACharacter`이기만 하면 되므로 **플레이어(`AMyPlayer`)와 동료 AI(`ACompanion`)가 똑같은 직업 컴포넌트를 그대로 공유**한다. 입력으로 구동하느냐 / AI로 구동하느냐만 다르고, 공격 로직은 한 벌만 둔다.

---

## 2. 직업 베이스 — UJobComponent

자식이 재정의하는 가상 함수는 3개다.

```cpp
virtual void Attack();                          // 자동공격 타이밍에 호출 (기본: 몽타주 재생)
virtual void OnAttackNotify(FName NotifyName);  // 몽타주 Notify 시점의 실제 타격/발사 판정
virtual void TickJob(float DeltaTime);          // 조준 무관 패시브용 (자가 회복 등). 기본 비어 있음
```

공통 데이터: `Damage`, `AttackInterval`(자동공격 간격), `EngageRange`(동료 AI 교전 사거리 — 근접은 짧게/원거리는 길게), `AttackMontage`, `AttackSound`, `WeaponMesh`.

베이스가 제공하는 **공용 헬퍼** 2개가 설계의 핵심이다.

- `EquipWeapon()` — 직업 무기를 플레이어 무기 슬롯에 끼운다(동료는 자기 BP 메시 사용).
- `SpawnProjectileForward(...)` — **전방으로 발사체를 스폰하고 직업의 Damage/속도를 적용**한다. → 궁수·마법사 공용.

---

## 3. 직업별 구현 (오버라이딩)

| 직업 | override | 방식 |
|------|----------|------|
| **검사**(`USwordsmanJob`) | `OnAttackNotify()` | 전방 **구체 스윕**(`SweepMultiByChannel`)으로 범위 내 적 동시 타격 + 넉백. `EngageRange=130`(붙어서 휘두름) |
| **궁수**(`UArcherJob`) | `Attack()` | `Super::Attack()`(몽타주) 후 `FireArrow()` → 베이스 헬퍼로 **화살(`AProjectile`) 1발 전방 발사**. 폭발 없는 단일 대상, 속도 2000 |
| **마법사**(`UMageJob`) | `Attack()` | 궁수와 같은 발사 헬퍼를 쓰되 스폰 후 `ExplosionRadius`만 설정 → **적중 지점 범위 폭발(AOE)**. 궁수보다 느리고(0.8s) 강함(Damage 2) |
| **힐러**(`UHealerJob`) | `Attack()` | "공격"이 곧 힐. 시전자 자신 + 전방 범위 아군 회복, 적(`AEnemy`)은 제외. 유일하게 적을 안 때리는 직업 |

> 포인트: **궁수/마법사는 "발사" 로직을 베이스 헬퍼(`SpawnProjectileForward`) 하나로 통일**하고, 차이(폭발 반경)만 자식에서 얹었다. 검사만 근접이라 `Attack()`이 아닌 `OnAttackNotify()`(타격 판정)를 재정의한 게 대비된다.

## 4. 전투 공통 베이스 — ACombatCharacter

"명색이 전투 캐릭터인데 왜 공격 함수가 없지?"가 헷갈려서 여기에 의도를 적어둔다.

`ACombatCharacter`는 **"공격하는 클래스"가 아니라 "전투에 참여하는(피해를 받고 죽는) 클래스"의 공통분모**다. 셋의 공격 방식이 전부 달라서 공통화할 "실제 공격"이 없기 때문에, 공격은 베이스에서 빼고 직업 축으로 분리했다.

```
ACombatCharacter (Abstract)  ← HP / 죽음 / Notify 배선
 ├── AMyPlayer    (입력으로 구동)
 ├── ACompanion   (AI로 구동)
 └── AEnemy       (직업 없이 자체 hit())
```

**베이스가 가진 것 = 공격이 아니라 "맞고 죽는" 쪽:**
`MaxHP / HP / Damage / IsDead` + `AddHP() / SetHP() / SetDead() / OnDeath() / OnRevive()`.
HP가 0 이하가 되면 `SetHP`가 자동으로 `IsDead`를 켜고, 죽음/부활 훅을 **전환 시점에 딱 1번씩** 호출한다(중복 방지). 검사가 적을 때릴 때 부르는 `enemy->AddHP(-Damage)`의 그 `AddHP`가 바로 이 베이스 함수다.

**공격에 대해 베이스가 하는 일은 "실행"이 아니라 "타이밍 배선"뿐이다.** 셋이 똑같이 쓰던 코드를 한 곳에 모았다.

```cpp
// ACombatCharacter::BeginPlay()
Anim->OnPlayMontageNotifyBegin.AddDynamic(this, &ACombatCharacter::OnMontageNotifyBegin);
// → 빈 가상 훅을 호출: "Notify 들어왔어, 실제 타격은 너희가 알아서"
virtual void HandleAttackNotify(FName NotifyName) {}
```

**자식이 재정의하는 가상 함수 3개** (`HandleAttackNotify` / `OnDeath` / `OnRevive`):

| 클래스 | HandleAttackNotify | OnDeath / OnRevive |
|--------|--------------------|--------------------|
| `AMyPlayer` | `CurrentJob->OnAttackNotify()` 위임 | — |
| `ACompanion` | `CurrentJob->OnAttackNotify()` 위임 | OnDeath만 (이동/콜리전 끄고 제거) |
| `AEnemy` | 자체 `hit()` (직업 없음) | OnDeath/OnRevive 둘 다 (오브젝트 풀 재사용) |

> 포인트: 플레이어와 동료는 `HandleAttackNotify`를 **직업 컴포넌트로 그대로 위임**(코드 동일)하지만, 적은 직업이 없어 **자체 `hit()`로 직접 타격**한다. 그래서 적은 풀 재사용 때문에 `OnRevive`까지 구현하고, 동료는 죽으면 사라지므로 `OnDeath`만 둔 차이가 있다.

---

## 정리

전투 코드를 **두 축**으로 나눈 게 이번 정리의 결론이다.

- **공격 방식** → `UJobComponent` (검 스윕 / 화살 / 파이어볼 / 힐)
- **체력·죽음** → `ACombatCharacter`

이 분리 덕분에 새 직업을 추가할 땐 `UJobComponent`만, 새 전투 캐릭터를 추가할 땐 `ACombatCharacter`만 건드리면 되고, 플레이어용으로 짠 코드가 동료 AI로 그대로 굴러간다.
